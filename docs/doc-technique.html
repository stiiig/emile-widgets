Documentation technique — Architecture & API

Mars 2026 · Next.js 15 · React 19 · Grist · N8N · GitHub Pages

1 Vue d'ensemble

EMILE est un ensemble de widgets React hébergés sur GitHub Pages. Ils s'intègrent dans Grist (mode iframe) ou fonctionnent en standalone via un proxy N8N (mode REST). Aucun backend applicatif dédié — la logique métier est répartie entre Grist (stockage), N8N (API proxy + HMAC) et le frontend (React/Next.js).

Browser (GitHub Pages stiiig.github.io) │ ├── Mode iframe ──▶ grist-plugin-api.js ──▶ Grist document (réseau interne) │ (injecté par Grist) │ └── Mode REST ────▶ N8N (accessible internet) │ vérifie HMAC, porte la clé API Grist └──▶ Grist REST API (réseau interne, inaccessible browser)

4 Proxy N8N & CORS

Les widgets tournent sur stiiig.github.io. L'API Grist est sur grist.incubateur.dnum.din.developpement-durable.gouv.fr. Deux raisons rendent l'appel direct impossible depuis le browser.

Problème 1 — CORS

La politique Same-Origin du navigateur bloque tout appel JS cross-domain sauf si le serveur répond avec Access-Control-Allow-Origin: *. L'instance Grist interne ne le fait pas. N8N s'intercale côté serveur : il appelle Grist sans contrainte cross-origin, puis renvoie la réponse avec le header CORS approprié.

Problème 2 — Clé API

L'API Grist REST exige Authorization: Bearer <clé>. Mettre la clé dans le bundle JS l'exposerait à quiconque ouvre les DevTools. La clé est stockée dans les credentials N8N (type Header Auth), jamais visible côté browser.

Architecture du proxy

Browser (stiiig.github.io) │ │ GET https://n8n.incubateur.dnum.../webhook/grist-proxy?table=CANDIDATS&token=… │ ← pas de clé API, pas de contrainte CORS côté browser ▼ N8N (accessible internet) │ 1. vérifie signature HMAC du token (credential EMILE_HMAC_SECRET) │ 2. ajoute Authorization: Bearer <clé de service Grist> │ 3. GET https://grist.incubateur.dnum.../api/docs/{docId}/tables/CANDIDATS/records ▼ Grist (réseau interne — inaccessible depuis internet) │ { "records": [{ "id": 42, "fields": {...} }] } ▼ N8N → ajoute Access-Control-Allow-Origin: * ▼ Browser → données disponibles dans React

Éviter le preflight CORS OPTIONS

Les requêtes simples (spec WHATWG Fetch) ne déclenchent pas de preflight. N8N ne gère pas la méthode OPTIONS. La règle dans rest.ts :

  • Lectures : GET → toujours simple
  • Écritures : POST avec Content-Type: text/plain;charset=UTF-8 → simple (pas de preflight)
  • Upload PJ : POST avec multipart/form-data → simple
  • ❌ Ne jamais utiliser Content-Type: application/json ni PATCH depuis le browser vers N8N
ℹ️
Le champ _action dans le body JSON sérialisé en text/plain permet à N8N de distinguer AddRecord ("_action":"add") de UpdateRecord ("_action":"update") dans le même workflow POST.

Clé de service Grist

Depuis mars 2026, N8N utilise un compte applicatif Grist dédié (clé de service) avec droits Éditeur limités au seul document EMILE. Le document est en accès restreint (Public access = Aucun accès). En cas de compromission, révoquer la clé dans Grist et la regénérer dans les credentials N8N — aucun changement de code frontend requis.

5 Tokens HMAC

Deux types de tokens sont utilisés : les tokens OCC (orienteurs) et le magic link admin (dashboard). Les deux reposent sur HMAC-SHA256 avec un secret partagé stocké uniquement dans N8N.

Token OCC — accès orienteur

token = rowId + "." + HMAC-SHA256(rowId.toString(), EMILE_HMAC_SECRET)

Exemple : "42.a3f8c2d1e9b7…"
           ↑               ↑
           rowId dans       signature sur 64 hex chars
           ACCOMPAGNANTS

Le token identifie de manière cryptographiquement vérifiée un enregistrement dans ACCOMPAGNANTS. N8N extrait le rowId avant le point, recalcule le HMAC et compare les deux signatures.

Magic link dashboard — accès admin 24 h

token = TS + "." + HMAC-SHA256(TS.toString(), EMILE_HMAC_SECRET)

Exemple : "1772798332.b4d9f1…"
           ↑                 ↑
           timestamp Unix    signature
           (secondes)

Expiration : N8N rejette si (Date.now()/1000 - TS) > 86400 (24 h)
⚠️
Secret unique. Les workflows occ-generate, occ-validate, occ-list, occ-get-candidat et le workflow dashboard utilisent tous le credential EMILE HMAC Secret (type Crypto dans N8N). Si une isolation renforcée est requise, utiliser deux credentials distincts.

6 Workflows N8N

8 workflows actifs. Tous exposés sur le même domaine N8N interne. Path webhook = dernier segment de l'URL de la variable d'env correspondante.

grist-proxy-get — lecture Grist GET

Variable : NEXT_PUBLIC_GRIST_PROXY_URL (partagée avec post)

Webhook GET /webhook/grist-proxy?table=X[&filter=JSON] ou ?attachId=Y │ ├─ attachId présent → GET /attachments/{id}/download → Respond Binary └─ table présent → GET /tables/{table}/records[?filter=...] → Respond JSON

Couvre : tables de données, tables de référence, métadonnées internes (_grist_Tables, _grist_Tables_column, _grist_Attachments).

grist-proxy-post — écritures + pièces jointes POST

Variable : NEXT_PUBLIC_GRIST_PROXY_URL

Webhook POST /webhook/grist-proxy │ ├─ Content-Type: text/plain (écriture JSON sérialisé) │ ├─ _action = "add" → POST /tables/{table}/records → { retValues: [newId] } │ └─ _action = "update" → PATCH /tables/{table}/records → {} │ └─ Content-Type: multipart/form-data (champ "upload") → POST /attachments → { data: "[42]" }

occ-generate — génération token OCC GET

Variable : NEXT_PUBLIC_OCC_GENERATE_URL

GET /webhook/occ-generate?rowId=X │ ├─ HMAC-SHA256(rowId, secret) → token = rowId.HMAC │ ├─ Respond { rowId, token, url } ← réponse immédiate au frontend │ └─ (background) PATCH ACCOMPAGNANTS.Lien_validation = url
💡
N8N répond avant le PATCH Grist. Le frontend reçoit le token sans attendre la persistance. Appelé par creation-compte-orienteur et inscription-candidat.

occ-validate — activation compte GET

Variable : NEXT_PUBLIC_OCC_VALIDATE_URL

GET /webhook/occ-validate?token=X.HMAC │ ├─ Vérifie HMAC → si invalide : { "status": "invalid" } │ └─ GET ACCOMPAGNANTS?filter={"id":[rowId]} ├─ Compte_valide = "Oui" → { "status": "already_validated" } └─ sinon → PATCH Compte_valide = "Oui" → { "status": "ok", "nom": "..." }

occ-list — liste des candidats GET

Variable : NEXT_PUBLIC_OCC_LIST_URL

GET /webhook/occ-list?token=X.HMAC │ ├─ Vérifie HMAC → si invalide : { "status": "invalid" } │ ├─ GET ACCOMPAGNANTS?filter={"id":[rowId]} → nom de l'orienteur │ └─ GET CANDIDATS?filter={"Responsable_candidat":[rowId]} → { "status": "ok", "orienteurNom": "…", "candidats": [...] }
⚠️
Responsable_candidat est une colonne Reference dans CANDIDATS → stocke un entier (rowId). Le filtre doit utiliser l'entier, pas le nom affiché. Ne jamais exposer le champ $Age calculé côté Grist — calculer l'âge côté frontend depuis dateNaissance.

occ-get-candidat — fiche candidat GET

Variable : NEXT_PUBLIC_OCC_GET_CANDIDAT_URL

GET /webhook/occ-get-candidat?token=X.HMAC&id=ROW_ID │ ├─ Vérifie HMAC token orienteur → si invalide : { "status": "invalid" } │ └─ GET CANDIDATS?filter={"id":[id]} → vérifie Responsable_candidat === rowId orienteur → { "status": "ok", "candidat": { tous les champs } }

occ-request-link — renvoi lien connexion POST

Variable : NEXT_PUBLIC_OCC_REQUEST_LINK_URL

POST /webhook/occ-request-link { email } │ ├─ Normalise email (minuscules) ├─ GET ACCOMPAGNANTS?filter={"Email":[email]} → rowId │ └─ non trouvé → { "status": "not_found" } │ └─ GET occ-generate?rowId=X → extrait token → reconstruit URL liste-candidats?token=X.HMAC (⚠️ pas l'url de occ-generate) → { "status": "ok", "url": "..." }

occ-request-validation-link — renvoi lien validation POST

Variable : NEXT_PUBLIC_OCC_REQUEST_VALIDATION_URL

POST /webhook/occ-request-validation-link { email } │ (identique à occ-request-link jusqu'à rowId) │ └─ GET occ-generate?rowId=X → utilise directement $json.url (pointe déjà vers validation-compte) → { "status": "ok", "url": "..." }
🔺
Différence critique entre les deux workflows de renvoi de lien :
occ-request-link doit reconstruire l'URL manuellement (cible : liste-candidats).
occ-request-validation-link utilise directement le champ url de occ-generate (cible : validation-compte).

7 Client REST (rest.ts)

src/lib/grist/rest.ts implémente l'interface GristDocAPI via le proxy N8N. Il est instancié par init.ts quand NEXT_PUBLIC_GRIST_PROXY_URL est défini.

Interface GristDocAPI

export type GristDocAPI = {
  fetchTable: (tableId: string) => Promise<any>;
  applyUserActions: (actions: any[]) => Promise<any>;
  getAttachmentDownloadUrl?: (attachId: number) => string;  // REST uniquement
  uploadAttachments?: (files: FileList) => Promise<number[]>; // REST uniquement
};

Mapping opérations → requêtes N8N

Appel frontendRequête N8NTraduit par N8N en
fetchTable("CANDIDATS") GET ?table=CANDIDATS GET /tables/CANDIDATS/records
fetchTable("_grist_Tables") GET ?table=_grist_Tables GET /tables/_grist_Tables/records
getAttachmentDownloadUrl(42) GET ?attachId=42 GET /attachments/42/download → binaire
uploadAttachments(files) POST multipart/form-data champ upload POST /attachments → [42]
applyUserActions([["AddRecord", "TABLE", null, fields]]) POST text/plain {"_action":"add","table":"TABLE","fields":{...}} POST /tables/TABLE/records → { retValues: [newId] }
applyUserActions([["UpdateRecord", "TABLE", id, fields]]) POST text/plain {"_action":"update","table":"TABLE","id":42,"fields":{...}} PATCH /tables/TABLE/records

Format de réponse Grist

// GET /tables/TABLE/records → format natif Grist
{
  "records": [
    { "id": 42, "fields": { "Prenom": "Alice", "Foyer": "Couple sans enfant", ... } },
    ...
  ]
}

// Types de valeurs Grist :
// - string, number, boolean, null
// - ChoiceList / RefList : ["L", val1, val2, ...]  ← sentinelle "L" + valeurs
// - Ref : entier (rowId dans la table référencée)
// - Date : entier Unix (secondes depuis epoch) — détection : 800_000_000 < v < 2_500_000_000
// - DateTime : float Unix (ex. 1772798332.28237)

11 Dashboard admin

Le dashboard est distinct des autres widgets. Il utilise un proxy N8N dédié avec vérification d'expiration 24 h, un tableau virtualisé pour gérer des milliers de lignes, et une résolution client-side des colonnes Ref.

Magic link dashboard

// Format token dashboard : TS.HMAC
token = timestamp_unix + "." + HMAC-SHA256(timestamp, SECRET)

// Flux côté client :
1. ?token=X dans l'URL → sessionStorage.setItem("db-emile-token", X)
2. history.replaceState() → URL nettoyée (token absent des logs serveur + historique)
3. Rechargement de page → token lu depuis sessionStorage
4. 401/403 N8N → sessionStorage effacé → écran "Lien expiré"

Tableau virtualisé (VirtualTable)

Seules les lignes visibles + un buffer de 6 lignes sont rendues dans le DOM. Cela permet d'afficher des milliers de lignes sans dégradation de performance. 

// Constantes de la virtualisation :
ROW_H  = 36  // px — hauteur fixe obligatoire
BUFFER = 6   // lignes rendues au-delà du viewport
COL_W  = 160 // px — largeur par défaut des colonnes
COL_MIN = 40 // px — largeur minimum après resize

// Calcul de la fenêtre visible :
startIdx = Math.max(0, Math.floor(scrollTop / ROW_H) - BUFFER)
endIdx   = Math.min(rows.length, Math.ceil((scrollTop + viewH) / ROW_H) + BUFFER)
// Seules rows[startIdx..endIdx] sont rendues dans le DOM

Résolution des colonnes Ref (refLookups)

Les colonnes Ref:TABLE de Grist stockent un entier (rowId). Pour afficher une valeur lisible, refLookups décrit quelle table charger et quelle colonne afficher. 

// Configuration dans TabDef :

12 Convertisseur Airtable → Grist

Outil de migration 100% client-side. Aucun serveur, aucune donnée ne transite hors du navigateur. Accessible sur /widgets/emile/airtable-to-grist/.

Architecture

src/app/widgets/emile/airtable-to-grist/
  ├── page.tsx      // UI React (3 onglets, upload, aperçu, download)
  ├── convert.ts    // Logique de conversion (import uniquement)
  └── styles.css

Fonctions de convert.ts

FonctionDescription
parseCSV(text)Parseur RFC 4180 complet. Gère les champs multi-lignes entre guillemets, le BOM UTF-8, les fins de ligne Windows/Unix.
toCSV(rows, headers)Sérialise en CSV avec guillemets sur tous les champs contenant virgule, guillemet ou saut de ligne.
convertEtablissements(rows)Remapping colonnes Airtable ETABLISSEMENTS → Grist ETABLISSEMENTS
convertAccompagnants(rows)Remapping colonnes Airtable ACCOMPAGNANTS → Grist ACCOMPAGNANTS
convertCandidats(rows)Remapping colonnes Airtable CANDIDATS → Grist CANDIDATS. Transformation des types Airtable (dates, cases, listes).

Cas particuliers de la conversion

  • Colonnes Ref Grist — stockent un rowId entier. Après import CSV, les colonnes Ref nécessitent une résolution manuelle dans Grist (les IDs Airtable ne correspondent pas aux rowIds Grist).
  • Dates Airtable — format ISO 8601 → converties en timestamps Unix (secondes) pour Grist.
  • ChoiceList Airtable — séparateur virgule → format Grist ["L", v1, v2].
  • Échantillon aléatoire — algorithme Fisher-Yates sur les lignes parsées, 30 lignes max.
🔒
Toute la conversion se fait en mémoire dans le navigateur via FileReader.readAsText(). Le fichier CSV n'est jamais envoyé à un serveur.
